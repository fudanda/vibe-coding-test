# ADR-0002：使用日志碎片降低多人协作冲突

- 日期：2026-06-23
- 状态：accepted
- 决策人：fudanda
- AI 协助：Codex
- 影响范围：多人协作流程、变更记录、PR 模板

## 背景

多人并行开发时，所有人都向同一个 `CHANGELOG.md` 追加内容，容易在同一日期、同一位置产生合并冲突。

## 决策

改用“一个有意义任务 = 一个变更文件”的日志碎片模式。`CHANGELOG.md` 只作为入口说明和索引规则，不再作为每次变更的追加文件。

## 采用方案

- 变更记录放在 `docs/vibe-coding/changes/YYYY/MM/`。
- 文件名包含日期、时间、作者和主题。
- 文件内容包含作者、Review、PR、Commit、影响模块、类型、原因、变更内容、验证和风险。
- 发布或阶段复盘时再按需汇总 `changes/`。

## 放弃方案

- 按人建目录：方便看个人记录，但弱化时间线，不利于发布复盘。
- 继续追加 `CHANGELOG.md`：简单但冲突频繁，不适合多人协作。
- 立即引入自动化脚本：当前阶段会增加维护成本，先用 Markdown 约定落地。

## 影响

- 每个需要记录的 PR 只新增自己的 change fragment 文件。
- PR 模板需要填写新增日志文件路径。
- 日常开发不强制维护汇总日志。

## 回看条件

如果碎片数量很多且发布复盘困难，再增加自动汇总脚本或 release note 生成流程。
